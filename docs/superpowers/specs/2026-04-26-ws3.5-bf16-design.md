## 1. Goal & Scope

### Goal

Add BFloat16 (BF16) as a first-class dtype variant alongside Float16, enabling end-to-end inference for BF16 ONNX models on the same surfaces FP16 just lit up.

### In-Scope Surfaces

- `Tensor::BFloat16(ArrayD<half::bf16>)` IR variant + parser path (ONNX `data_type = 16`).
- GPU dispatch infrastructure: `DType::BFloat16`, `GpuTensor::zeros_bf16`, memory pool BF16 arm, `upload_initializers` BF16 path.
- Kernel-level dispatch arms (memcpy / elementwise / reductions / Cast) at every existing `DType::Float16 =>` site.
- Cast extensions: FP32↔BF16, FP16↔BF16 (FP16↔BF16 pair is net-new — manual checklist item, not surfaced by R3 wildcard removal).
- MatMul + Conv via garboard `gemm_bf16` (im2col + GEMM mirror of M3.6's FP16 path; mechanical addition applies to both `gpu_conv1d` and `gpu_conv2d`).
- Bool cross-roster verify: no new Bool work — existing Bool surface already passes; verification only. Where BF16 dispatch arm additions land at sites with existing "Bool dispatch arm pending" markers (5 ops noted at WS-3 close: CumSum, Transpose, gpu_copy, Concat, 2× Slice), the Bool markers stay deferred. Closing those Bool gaps is a separate follow-up if/when a roster model exercises them.

### Roster Commit

- **Primary integration targets:** BERT-base BF16 (matmul-only canary) + Whisper-Tiny BF16 (Conv1D + attention + Cast-at-boundary stress).
- **Y(5) regression-sweep canaries:** Whisper-FP16 (M3.7c sign-off), DistilBERT-INT8 (M4.7 sign-off, seq_len ≤ 8 caveat per memory), GPT-2 (M3.7b regression sentinel), plus all M2.x roster members already gated.

### Known Coverage Gap (Documented, Accepted)

Conv2D BF16 ships unvalidated empirically. `gpu_conv2d` and `gpu_conv1d` share the im2col + GEMM lowering, so the BF16 dispatch arm addition is the same mechanical change for both — but the roster (BERT no Conv, Whisper-Tiny encoder Conv1D only) does not exercise Conv2D. Risk acknowledged. Adding ResNet-50 BF16 to the roster would close the gap empirically but expands WS-3.5 scope; deferred to a follow-up workstream.

### Explicit Non-Goals

- T5-small BF16 (encoder-decoder Cast surface; stretch goal status, revisit only if BERT + Whisper land with bandwidth).
- Native batched-BF16 GEMM (per-batch loop over `gemm_bf16` acceptable; tracked as Follow-Up #2).
- Direct cuDNN BF16 conv binding (im2col + `gemm_bf16` is the WS-3.5 path; cuDNN-direct stays a future perf milestone).
- ResNet-50 BF16 (would close Conv2D coverage gap; deferred).
- Kernel-file unification (`kernels_fp16.rs` + `kernels_bf16.rs` → `kernels_half.rs` — Follow-Up #1, not WS-3.5 work).
- Broader silent-fallthrough sweep (WS-1 M1.5/M1.6 domain, not WS-3.5).
- DistilBERT-INT8 numerical divergence at long seq_len (separately tracked iconnx investigation; memory-noted 2026-04-26).
- Kokoro audio output shape mismatch at seq_len=50 (separately tracked iconnx investigation; memory-noted 2026-04-26).

## 2. Architecture

WS-3.5 follows a strict additive-dispatch-arm architecture: BF16 is a new dtype variant added everywhere FP16 already exists, with no structural refactoring.

### Type Plumbing

`half::bf16` (already a transitive dependency via the `half` crate, used by garboard's `gemm_bf16`) is the IR-level scalar type. `Tensor::BFloat16(ArrayD<half::bf16>)` mirrors `Tensor::Float16(ArrayD<half::f16>)`. CUDA-side `__nv_bfloat16` (from `<cuda_bf16.h>`) is bit-compatible with `half::bf16` (both `repr(transparent)` over a 16-bit word) — kernel launches pass `*const half::bf16` as `*const __nv_bfloat16` via the same pointer-cast pattern FP16 uses.

### Dispatch Infrastructure

`DType::BFloat16` enum variant. Sized to 2 bytes parallel to `DType::Float16`. Memory pool gains a BF16 arm. `GpuTensor` gains `zeros_bf16` constructor and `data_bf16()` accessor. `upload_initializers` gains the BF16 host-side branch in `upload_tensor`.

### Kernel Files

`src/cuda/ops/kernels_bf16.rs` (new file, mirrors `kernels_fp16.rs`) hosts BF16-specific NVRTC source strings and launcher functions. NVRTC source includes `#include <cuda_bf16.h>` and uses `__nv_bfloat16` / `__nv_bfloat162` types parallel to `__half` / `__half2`. CUDA bf16 intrinsics (`__hadd`, `__hmul`, `__hsub`, `__bfloat162float`, `__float2bfloat16`) substitute for fp16 equivalents. A second new file `src/cuda/kernels/kernels_bf16.rs` mirrors `src/cuda/kernels/kernels_fp16.rs` for the kernels module (broadcast-scalar, etc.).

### Reduction Strategy

F32 accumulator, mirroring FP16 M3.4 sub-3. BF16 has wider exponent range than FP16 but narrower mantissa (7 bits vs 10). Naive in-precision accumulation overflows less but rounds more. Pattern: `__bfloat162float` lift before accumulation, store back via `__float2bfloat16`.

### MatMul / Conv Routing

`gpu_matmul_2d` adds a `DType::BFloat16` arm calling garboard `gemm_bf16` (CUBLAS_COMPUTE_32F single-flavor — no acc16 variant exists in cuBLAS API). `gpu_batched_matmul` and `gpu_matmul_2d_3d` fall back to per-batch loop over `gemm_bf16` (parallel to FP16's per-batch loop; correctness-clean, perf-suboptimal until Follow-Up #2). `gpu_conv1d` and `gpu_conv2d` already lower as im2col → GEMM; once their inner `match input.dtype()` accepts BF16 routing to `gemm_bf16`, the conv pipeline lights up automatically (same automatic-lighting pattern M3.6 used for FP16).

### `gpu_copy` / Identity

*Not polymorphic.* Per WS-1 M1.4 commit `01213c5`, dispatch is per-dtype with explicit kernels (`copy_kernel` f32, `copy_i64_kernel`, `copy_i32_kernel`). BF16 needs an explicit `copy_bf16_kernel` plus a dispatch arm — same shape Float16 needed. (Note: this site is one of the 5 carrying a "Bool dispatch arm pending" marker per WS-3 sign-off; BF16 work touches it, but Bool stays deferred per the discipline call in §1.)

### Cast Routing

Four categories of dtype-pair arms (10 individual arms total):

- `(BFloat16, Float32)` and `(Float32, BFloat16)` — direct lift/round via `__bfloat162float` / `__float2bfloat16` in NVRTC kernel.
- `(BFloat16, Float16)` and `(Float16, BFloat16)` — go through f32 intermediate. ONNX spec doesn't define direct half-half cast; iconnx avoids precision-coupling between the two half formats.
- `(BFloat16, Int32)`, `(Int32, BFloat16)`, `(BFloat16, Int64)`, `(Int64, BFloat16)` — direct via `__bfloat162int` / `__int2bfloat16` family or f32 intermediate.
- `(BFloat16, Bool)` and `(Bool, BFloat16)` — explicitly unsupported, parallel to FP16↔Bool stance. No active roster graph routes this; per M3.7b discipline, dispatch arms added only when exercised, with cross-roster verify at the same time.

Audit confirmed by grep: existing FP16 Cast surface today is FP16↔{FP32, Int32, Int64} supported + FP16↔Bool explicitly rejected. **No FP16↔Int8 or FP16↔UInt8 arms exist** (WS-4 INT8 quantization chains stay in their own dtype-pair surface). BF16 mirrors exactly this set.

### Y(1) dtype-pair Op Audit

One-time grep at Y(1) for ops beyond Cast that dispatch on `(DType, DType)` patterns or per-pair kernel tables. Candidates to check:

- GEMM dtype-combo selectors (M4.6 MatMulInteger 4-kernel pattern).
- Reductions with `(input_dtype, accumulator_dtype)` selection.
- LayerNorm / Softmax with `(input_dtype, intermediate_dtype)` selection.

Audit result documented in spec writeup at Y(1) — even if "Cast is the only one," the audit having happened is the point. Each non-Cast site found becomes an explicit checklist item alongside Cast's 10 arms.

### Boundary Policy

ONNX models with BF16 inputs/outputs follow the same Cast-at-boundary pattern Whisper-FP16 uses (per M3.5). Graph inputs cast BF16 → FP32 at executor entry; graph outputs cast FP32 → BF16 at executor exit. This is the `gate_fusion_to_float32` policy applied as-is — WS-3.5 only extends the dtype audit in that helper, doesn't change the policy.

### Test Fixture Policy

Per `feedback_dtype_test_fixtures` memory: all numerical-compare gates use Box-Muller seeded random fixtures.

- BERT-base BF16: random `input_ids` over vocab range + random `attention_mask` distribution (not all-ones).
- Whisper-Tiny BF16: re-uses `whisper_encoder_fp16_features` Gaussian mel pattern from leadline-bench `de2829e`, regenerated for BF16 dtype.

Tolerance for both: empirically measured by leadline-bench post-integration (NOT assumed-from-FP16). BF16's 7-bit mantissa vs FP16's 10-bit gives more per-op rounding while wider 8-bit exponent gives less catastrophic cancellation; net rel-diff vs ORT likely similar (~0.3–1%) but measured not assumed.

## 4. Data Flow

WS-3.5 introduces no new data-flow stages. Existing FP16 data flow extends transparently to BF16 because every stage that handles FP16 will handle BF16 via parallel dispatch arms.

### Stage 1: ONNX Parse → IR

ONNX file with BFLOAT16 (`data_type = 16`) initializers / value_infos → `onnx_parser.rs::parse_tensor_proto` returns `Tensor::BFloat16(ArrayD<half::bf16>)`. Same path Float16 took at WS-3 M3.2: read `raw_data` as `&[u8]`, transmute pairs to `half::bf16` (`repr(transparent)` over a 2-byte word), construct `ArrayD`. Endianness is little-endian per ONNX spec, matching x86_64. IR consumers (shape inference, fusion detection, lowering) operate on shape + dtype-tag, not on the scalar bytes — no change required.

### Stage 2: IR → ExecutionPlan (Lowering)

Fusion gating: `gate_fusion_to_float32` extension inspects BF16 initializers + Constant nodes to suppress f32-only fused kernels when BF16 inputs would route through them. Identical logic to the FP16 gating from M3.7. Memory plan: `is_planner_eligible` (Phase 4) treats BF16 like Float16 — eligible for arena allocation, sized at 2 bytes per element, considered for power-of-2 size-class slots ≥ 4096.

### Stage 3: Initializer Upload (Host → Device)

`executor::upload_tensor` matches on `Tensor::BFloat16(arr)` and copies `arr.as_slice().unwrap()` into a `GpuTensor` allocated from `memory_pool.get_tensor_bf16`. Mirror of M3.7a Float16 arm. No transformation: `half::bf16` and `__nv_bfloat16` share bitlayout; copy is raw memcpy.

### Stage 4: Per-Op Execution

Executor dispatch table receives a `GpuTensor` with `dtype = DType::BFloat16` and routes to per-op kernels. For each in-scope op, the op's internal `match input.dtype()` includes a BF16 arm calling either:

- A direct BF16 NVRTC kernel from `kernels_bf16.rs` (memcpy-class, elementwise, comparison).
- A reduction kernel with f32-accumulator pattern (LayerNorm, Softmax, ReduceMean, etc.).
- A garboard call (`gemm_bf16` for MatMul / Conv via im2col).
- A Cast lift via the new dtype-pair arms in `cuda/ops/cast.rs`.

Mid-graph Cast nodes routing to/from BF16 use the `cast.rs` dispatch arms from §2 — no boundary-special-case needed; the boundary cast in Stage 5 is the same Cast dispatch applied at executor entry/exit, not a separate code path.

### Stage 5: Boundary Cast at Executor Entry / Exit

- BERT-base BF16: graph inputs `int64` token IDs + `int64` attention mask (no cast); computation tensors BF16 internally; logits output BF16 → cast to FP32 via `cast.rs (BFloat16, Float32)` arm before host return.
- Whisper-Tiny BF16: graph input FP32 mel features (Box-Muller seeded) → cast FP32 → BF16 at conv boundary via `cast.rs (Float32, BFloat16)`; encoder runs entirely in BF16; last_hidden_state BF16 → cast BF16 → FP32 before host return.

iconnx never auto-inserts dtype casts for mixed-dtype operations. The M3.6 typed-error stance ("no silent f32 fallthrough") extends to BF16: any operation with BF16 × FP16 / BF16 × FP32 inputs returns a typed dtype-mismatch error rather than silently coercing. Mixed-dtype operations require explicit `Cast` nodes in the ONNX graph.

iconnx performs BF16 → FP32 cast at executor exit before host return. leadline-bench's existing `extract_ort_f32_outputs` and iconnx `as_slice()` paths work unchanged because outputs arrive as FP32. leadline-bench requires no BF16 awareness.

### Stage 6: Compare Gate

leadline-bench loads ORT FP32 reference outputs + iconnx BF16 outputs (cast back to FP32 at executor exit per Stage 5). Tolerance via `dual(abs, rel)` per leadline-bench profile (empirically measured for `BertBaseBf16` and `WhisperEncoderBf16`, NOT inherited from FP16). Y(5) full-roster sweep also re-runs Whisper-FP16 (M3.7c sign-off), DistilBERT-INT8 (M4.7 sign-off, seq_len ≤ 8 caveat per memory), GPT-2 (M3.7b regression sentinel), plus all M2.x roster members already gated. Gate is "no regression vs prior sign-off."

No new asynchronous interactions. All BF16 work runs on the same stream the rest of the executor uses; no parallelism / synchronization changes.

## 5. Error Handling

WS-3.5 error handling extends the discipline established by WS-3 M3.6 (typed errors), M3.4 sub-2 (no `_ =>` wildcards on dispatched dtypes), and WS-1 (typed parser errors). No new error categories; just BF16-aware extensions to existing ones.

### Parser Errors (Y(1))

- BFLOAT16 (`data_type = 16`) initializer with `raw_data.len() % 2 != 0` → `ParseError::MalformedInitializer { name, dtype: "bfloat16", reason: "raw_data length not multiple of 2" }`. Same pattern as Float16 path landed at WS-3 M3.2.
- BFLOAT16 value_info with mismatched element count → `ParseError::MalformedValueInfo { ... }`.
- The two existing test fixtures `bf16_weights` and `empty_bf16` flip from expecting `UnsupportedInitializerDType` / `UnsupportedDType` to expecting successful parse with `Tensor::BFloat16(...)`. Tests rewritten as part of Y(1).

### Hardware Support Errors (Y(1), NEW Subsection)

- BF16 native compute requires `sm_80+` (Ampere and later: A100, H100, RTX 30/40-series). Older Turing (`sm_75`: RTX 20-series, T4) compiles bf16 NVRTC but emulates with significant perf loss; some intrinsics (`__bfloat162float`, `__hadd` on `__nv_bfloat16`) require `sm_80+` outright. Volta (`sm_70`, V100) has no BF16 support.
- WS-3 FP16 was `sm_70+`; WS-3.5 BF16 native is `sm_80+` — meaningfully tighter floor.
- Detection seam: garboard's `device.props.compute_capability: (major, minor)` (per `cuda/device.rs:144`) — already plumbed for NVRTC `--gpu-architecture` flag. iconnx reads via garboard device handle.
- Enforcement seam: `GpuTensor::zeros_bf16` (single ctor entry for any BF16 GpuTensor) checks `compute_capability >= (8, 0)`; if not, returns typed `OpError::UnsupportedHardware { dtype: "bfloat16", required_compute_capability: "sm_80", device_compute_capability: "sm_75" }`. Defense-in-depth principle: every code path that touches BF16 GpuTensors hits this single ctor; downstream dispatch sites can assume BF16 GpuTensor exists ⇒ hardware capable.
- `executor::upload_tensor` BF16 arm calls `zeros_bf16`, so initializer-driven BF16 graphs surface the error at executor entry — caught early in the load path, not at first dispatch call.
- The hardware check is one-time-per-graph-load (not per-op-call); zero perf overhead.

### Dispatch Errors (Y(2), Y(3))

- Unsupported dispatch arm (e.g., a non-roster op gains a BF16 input but no BF16 implementation) → typed `OpError::UnsupportedDtype { op, dtype: "bfloat16" }`. Loud rather than silent.
- Per §2, R3 wildcard removal at FP16-touched sites guarantees compile-time enumeration: cargo build fails until every BF16 site has either a real arm or an explicit `DType::BFloat16 => Err(UnsupportedDtype { ... })` arm. No `_ =>` silent fallthrough on the FP16 surface.
- Sites *outside* the FP16 surface keep their existing wildcard arms (per R3 hybrid bound — not a full silent-fallthrough sweep). Non-FP16-surface dispatch sites mostly use typed-error wildcards per WS-1 progress to date; remaining `panic!`-on-unexpected-dtype paths (if any) are tracked in WS-1's M1.5/M1.6 punch list, not in WS-3.5 scope. WS-3.5 doesn't introduce new panic paths and doesn't fix existing ones outside the FP16 R3 surface.

### Cast Errors (Y(2))

- `(BFloat16, Bool)` and `(Bool, BFloat16)` casts return `OpError::UnsupportedCast { src, dst }` (M3.7b discipline — exercised arms only).
- Direct half-half casts go through f32 intermediate; never panic on a precision-coupling assertion.
- Mismatched dtype-pair (e.g., Cast op claiming output BF16 but graph says FP32) → typed graph-validation error at lowering, not at runtime.

### MatMul / Conv Errors (Y(3))

- BF16 × FP16, BF16 × FP32 mixed-dtype matmul → typed `OpError::DtypeMismatch { op: "MatMul", lhs, rhs }`. Never silent f32 fallthrough.
- Grouped Conv + BF16 (`gpu_conv1d` doesn't support groups, parallel to FP16 stance) → typed `OpError::UnsupportedDtype { op: "Conv", dtype: "bfloat16", reason: "groups != 1" }` with explicit reason field.
- garboard `gemm_bf16` runtime errors (CUDA / cuBLAS underlying failure) → typed `OpError::CudaError(garboard::Error)` propagation. Same path FP16 already uses.

### Reduction Overflow Strategy (Y(2))

- Practical overflow risk is nil for roster targets (Whisper K=1500, BERT K=768 — overflow would require per-element magnitudes >1e10, far out of normalized network distribution).
- If overflow ever occurred, ±Inf or NaN would propagate to outputs silently (CUDA does not natively trap on FP overflow — there is no kernel-internal trap). Detection is downstream-side: surface at the leadline-bench compare gate as a numerical mismatch.
- No silent recovery; gate catches it. Honest framing: detection is gate-side, not kernel-side.

### Boundary Cast Errors (Y(4), Y(5))

- Model declares BF16 output but executor's exit-cast helper has no `(BFloat16, Float32)` arm → graph-validation error at executor construction (per the Y(1) hardware-capability check seam, this is now a load-time check), not at first inference call. Caught early.
- Mel features wrong dtype (e.g., FP16 input fed to BF16 graph) → typed dtype-mismatch error at first dispatch call (no discrete graph-validation pass for input-binding today).

### Test Failure Modes (Y(4), Y(5))

- Numerical gate failure (max_abs_diff > tolerance) is a separable failure class from structural failure (panic / typed error). The Y(5) sweep uses `dual(abs, rel)` per profile and reports each model individually — no model-failure-aborts-sweep behavior.
- Per `feedback_dtype_test_fixtures` memory, numerical failure on Box-Muller seeded random fixtures is a real correctness signal (not test methodology); apply the M3.7 numerical-bisect probe pattern (`run_with_checkpoints` in `cuda/inference/mod.rs`) before tolerance-relaxing.

### Plan-Oversight Discipline (Cross-Cutting)

- No `_ =>` wildcard arms introduced in WS-3.5. R3's wildcard-removal-on-FP16-surface is the explicit application; non-FP16 surfaces stay out of scope.
- Every typed error has a structured reason field (parallel to M3.6 typed errors) — not just an opaque error string.
- WS-1 typed `ParseError` taxonomy stays the source of truth for parser errors; WS-3.5 only adds new variants where genuinely new (`bfloat16` dtype tag in existing variants, not new error variants).

### Graph-Validation Pass Status

*No discrete graph-validation pass exists between IR construction and execution today.* `ir/passes/` contains shape_inference, fusion, memory_planner, etc. — but no validate.rs. Errors surface at first inference call via the typed errors at each dispatch site. Cast errors, mid-graph dtype mismatches, and unsupported-op errors are all dispatched lazily.

Two new exceptions land in WS-3.5 at executor construction (load-path, not first-dispatch): (a) hardware-capability check in `GpuTensor::zeros_bf16`, (b) BF16 initializer parse / upload. Both are graph-load-time so they fail fast vs first-inference timing.

The hardware-capability check could seed a future broader graph-validation pass, but **creating that pass is out of WS-3.5 scope.** Tracked as a possible future architecture-pass workstream item (parallel to the deferred line-count debt cleanup).

### No Silent Fallback Policy

WS-3.5 explicitly does not introduce any "if BF16 fails, retry as FP32" or "downcast and warn" recovery paths. Either the BF16 path works correctness-stable per its dispatch table, or it errors loudly. This matches WS-3's stance and Ryan's CLAUDE.md "Principled solutions only. NO HACKS" instruction.

## 6. Testing Strategy & Sign-Off Gates

WS-3.5 testing reuses WS-3's gate structure with BF16 dispatch added.

### Y(1) — IR + Parser + Dispatch Infra

- Unit tests in `src/tensor.rs`: extend the "Float16 + Bool surface audit" tests to a "Float16 + BFloat16 + Bool surface audit" — parallel test cases for BF16 panic-on-wrong-accessor pattern.
- Unit tests in `src/onnx_parser.rs`: flip `bf16_weights` and `empty_bf16` fixtures to expect successful parse; add malformed-bf16 fixtures parallel to FP16's malformed cases.
- Unit test for hardware-capability check at `GpuTensor::zeros_bf16` — synthetic test that injects a fake compute capability < (8, 0) and asserts typed `UnsupportedHardware` error.

### Y(2) — Kernel Dispatch Arms

- Per-op unit tests parallel to FP16's: `tests_bf16` modules under each op directory (mirrors `tests_fp16.rs` under `cuda/ops/layout/`).
- Numerical correctness tests for memcpy / elementwise / reductions — Box-Muller seeded random fixtures, BF16 vs FP32-reference comparison via host roundtrip.
- Cast-pair tests for all 10 new dtype-pair arms (FP32↔BF16, FP16↔BF16, Int32/Int64↔BF16); explicit-unsupported tests for (BFloat16, Bool) pair.
- R3 wildcard-removal verification: cargo build + clippy with `-D warnings` after wildcard removal, before BF16 arm addition. Compile-error punch list captured. Commit ordering: punch list → arms in same commit (unified surface boundary per Y(2)'s leadline-confirmed "compression target, not hard cap").

### Y(3) — MatMul + Conv

- `gpu_matmul_2d` / `gpu_batched_matmul` BF16 unit tests against host f32 reference via roundtrip cast. K=1500 stress (Whisper attention shape) + K=768 (BERT attention) + small K (correctness).
- `gpu_conv1d` BF16 unit tests against host f32 reference. Conv2D BF16 mechanically lit but unvalidated empirically (§1 acknowledged gap).
- Typed dtype-mismatch error tests: BF16 × FP16 matmul, BF16 × FP32 matmul, grouped + BF16 conv.

### Y(4) — BERT-base BF16 e2e

- New `tests/bert_base_bf16_test.rs` structural-load test: model parses, executor construction succeeds on test hardware, first inference doesn't panic.
- leadline-bench `BertBaseBf16` ModelProfile gate: numerical compare vs ORT FP32 with empirically-measured `dual(abs, rel)` tolerance. leadline measures and locks tolerance during integration; iconnx side accepts whatever leadline-bench gates with.
- Box-Muller seeded random `input_ids` + random `attention_mask` distribution (not all-ones).

### Y(5) — Whisper-Tiny BF16 e2e + Roster Sweep

- New `tests/whisper_encoder_bf16_test.rs` mirrors `whisper_encoder_fp16_test.rs`. Structural-load test + e2e gate.
- leadline-bench `WhisperEncoderBf16` ModelProfile gate. Box-Muller seeded mel features regenerated for BF16 dtype. Empirically-measured tolerance.
- **Roster regression sweep:** all 8 active roster models run end-to-end. Gate: no regression vs prior sign-off thresholds. Specifically:
  - Whisper-FP16: prior sign-off `abs 0.039 / rel 0.30%` (per WS-3 close).
  - DistilBERT-INT8: prior sign-off `abs 0.005` at seq_len ≤ 8 (per memory caveat).
  - GPT-2: must continue running clean (M3.7b regression sentinel).
  - All others: must remain at-or-below their prior tolerance.
- **Kokoro and YOLOS-tiny element-wise gates** are disabled per leadline-bench commit `10aa598` (audio waveform / detection bounding boxes are not element-wise reproducible across backends; semantic comparison is tracked as future leadline-bench work). Both will SKIP rather than fail in the roster sweep — a SKIP is not a regression. New tolerance failures or worse magnitudes for previously-clean models DO count as regressions.
