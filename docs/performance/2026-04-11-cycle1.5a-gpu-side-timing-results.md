## Headline

Added per-op **GPU-side** kernel timing via a new `GpuEventTimer` helper. `ProfileData` now has a `gpu_op_times` map that populates alongside the existing `op_times` during `run_with_profiling`. The live Kokoro run confirms all 52 keys match between the two maps, and reveals which ops are enqueue-bound vs compute-bound — exactly the diagnostic Leadline needs to decide the next optimization target.

### Column semantics (critical — see spec-bug note below)

| Column | Measurement | Source |
|---|---|---|
| `cpu_enqueue` | CPU-side async enqueue time — how long the dispatch call took to queue the kernel | `Instant::now()` wrapping `execute_gpu_operator_with_precomputed()`, which is async (fire-and-forget) |
| `gpu_kernel` | GPU-side wall time of the actual kernel execution on the stream | `CudaEvent::elapsed_ms` between paired events recorded before/after dispatch |

These are **orthogonal measurements**. The earlier draft of the spec incorrectly treated `op_times` as CPU wall-clock and proposed `launch_overhead = cpu - gpu`. That formula is wrong: `op_times` IS the launch overhead directly (dispatch is async, so the elapsed time only covers the enqueue call). This is documented and corrected in the Cycle 1.5a Task 5 commit message.

## Interpretation

The 10-row table cleanly separates three classes of operator:

### Enqueue-bound (fusion / graph capture would help)

**`Fused_DivRsqrt`: 73,167 cpu_enqueue vs 24,110 gpu_kernel — 3x more time enqueuing than executing.**

This is striking: the kernel itself is fast (24 ms across 65 invocations = ~370 μs/call), but dispatching each call costs ~3x that. The fused op is effectively a driver-thrash operation. Candidates for fixing:
- CUDA graph capture to batch the 65 dispatches into one submission.
- Consolidating adjacent `Fused_DivRsqrt` calls into larger batches at the graph level (if the detection walker can identify them).
- The existing fusion already reduces launches; further reduction needs structural changes.

**`Fused_MulSinPowMulAdd`: 31,496 cpu_enqueue vs 25,555 gpu_kernel — slightly enqueue-dominant.**

At 1.2x ratio, the new fusion pattern from Cycle 1 is close to balanced. The fused kernel is doing real work (25 ms / 48 calls = ~530 μs/call, which is substantial for a 5-op chain), and the enqueue is proportionally large. This is near the optimal fusion sweet spot — more fusion on this chain wouldn't help much.

### Balanced (close to 1:1 enqueue-to-compute)

**`Conv`: 122,301 cpu_enqueue vs 131,592 gpu_kernel** — 0.93 ratio. Conv kernels are heavy enough that enqueue overhead is roughly matched by compute. Fusion is unlikely to help here; kernel tuning or cuDNN algorithm selection might.

**`Gemm`: 39,420 vs 40,735** — 0.97 ratio, balanced.
**`MatMul`: 13,188 vs 15,598** — 0.85 ratio, balanced.
**`LSTM`: 35,116 vs 40,507** — 0.87 ratio, balanced. (Interesting: Cycle 1's plan mentions cuDNN LSTM as a future optimization, which would replace this with a fused `cudnnRNNForward` call.)

### Compute-bound (kernel optimization would help)

**`ReduceMean`: 3,883 cpu_enqueue vs 50,828 gpu_kernel — 13x MORE kernel than enqueue.**

`ReduceMean` is the clear outlier: 130 dispatches, each of which does substantial GPU work relative to its dispatch cost. Fusion or graph capture won't help; the kernel itself is the bottleneck. Possible improvements:
- Use cuDNN or cuBLAS reduction primitives (if applicable to the tensor shapes).
- Hand-tune the kernel for the common Kokoro shapes (per-channel, per-feature).
- Algorithmic: fuse with an adjacent op (e.g., `ReduceMean → LayerNormalization` is a known pattern).

**`Add`, `Mul`, `Sub`** all have kernel time > enqueue time (~1.3–1.5x ratio). These are many small elementwise ops; fusion reduces launches but each call's kernel is still cheap. The fusion work in Cycle 1 already covered the biggest elementwise chain.

## Prioritization signal for Cycle 2

Based on this data, the highest-impact optimization targets are (in rough order):

1. **`Fused_DivRsqrt` enqueue optimization.** 73 ms of dispatch time for only 24 ms of actual work — the biggest enqueue-overhead surplus in the whole profile. 3x ratio. Either CUDA graph capture or consolidating the 65 calls into fewer larger dispatches would reclaim 30–50 ms of wall clock. This alone could match Cycle 1's entire 46 ms improvement.
2. **`ReduceMean` kernel tuning or fusion with adjacent ops.** 51 ms of compute — the second-largest GPU kernel cost after Conv. Fusion with `LayerNormalization` (31 calls) is a known pattern and would eliminate 130 - 31 = 99 separate reduction launches.
3. **`Conv` algorithm selection.** 132 ms of kernel time across 88 calls is ~1.5 ms/call. cuDNN has multiple algorithms for each Conv shape; iconnx may not be picking the best one. Worth measuring.
4. **`AddMulAdd` detection gap** (the Cycle 1 results doc already flagged this as 51 instances detected by Leadline but 0 matched by the existing walker). Independent of this data.
5. **cuDNN LSTM replacement** (40 ms kernel time, balanced ratio). Still worth doing — `cudnnRNNForward` fuses the 4 gate kernels into one and eliminates several dispatches.

Note: the `Fused_DivRsqrt` surprise (enqueue-bound at 3x) overtakes the originally-planned AddMulAdd detection work in priority. Leadline can decide based on this whether to pivot Cycle 2.

### `GpuEventTimer` helper API

```rust
pub struct GpuEventTimer { /* events, labels, ctx, stream */ }

impl GpuEventTimer {
    pub fn new(ctx: Arc<CudaContext>, stream: &Arc<CudaStream>, estimated_ops: usize) -> Result<Self, CudaError>;
    pub fn record(&mut self, label: String) -> Result<(), CudaError>;
    pub fn finalize(self) -> Result<BTreeMap<String, (u64, usize)>, CudaError>;
}
```

- All events created with `Some(CUevent_flags::CU_EVENT_DEFAULT)` — cudarc's default is `CU_EVENT_DISABLE_TIMING`, which silently breaks `elapsed_ms`.
- Single `stream.synchronize()` call in `finalize()`, not per-op — async execution is preserved.
- Zero-elapsed entries are still inserted into the map (preserves key-parity invariant with `op_times`).
- Errors propagate via `CudaError::Kernel` with descriptive messages. No silent fallback.

### Integration in `run_with_profiling`

Ten-line surgical diff at five edit sites:
1. Import (`use self::gpu_event_timer::GpuEventTimer;`)
2. Timer construction before the execution loop
3. `gpu_timer.record(pattern_name.to_string())?` at the fused-pattern dispatch site
4. `gpu_timer.record(node.op_type.clone())?` at the regular-op dispatch site
5. `profile.gpu_op_times = gpu_timer.finalize()?` after the loop

**Key-parity invariant enforced structurally**: both `record()` calls use the *exact same variable* already used for `op_times.entry(...)` — no `format!`, no second match, no helper function. A reviewer can visually confirm pairing.

## Spec correction

The original spec (committed at `0db44cd6` on 2026-04-11) incorrectly described `op_times` as CPU wall-clock time and proposed a `launch_overhead = cpu_time - gpu_time` formula. During Task 5 execution, the implementing agent discovered that iconnx's `op_times` is explicitly annotated as async enqueue time in `src/cuda/inference/mod.rs` (`// Time the operation (async - measures enqueue time, not execution time)`), making the formula backwards. The Task 5 commit (`113147af`) drops the bogus "GPU ≤ CPU + slack" assertion, reframes the printout as "Top 10 by GPU kernel time" with cpu_enqueue and gpu_kernel side by side, and rewrites the `ProfileData::gpu_op_times` doc comment to correctly describe the orthogonal semantics.

The spec document itself (`docs/superpowers/specs/2026-04-11-gpu-side-kernel-timing-design.md`) still has the original incorrect framing and should be amended in a follow-up commit — but the code is correct, and all tests pass.
